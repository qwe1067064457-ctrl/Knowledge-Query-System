# 开发环境启动与前端联调经验

## 1. 本次现象

本次前端测试变慢，主要不是因为缺少 plugin 或 skill，而是因为本地开发链路本身较重，同时叠加了权限审批和冷启动成本。

## 2. 主要原因

1. 启动本地进程时，可能会遇到权限审批或系统限制，导致服务没有立刻拉起。
2. 前端使用 `next dev -p 3000`，首次启动会进行开发态编译。
3. 后端使用 `uvicorn --reload`，除了服务本身，还会额外启动文件监听。
4. 后端启动时还会执行初始化逻辑，例如：
   - `refresh_snapshot(settings.backend_dir)`
   - `agent_manager.initialize(settings.backend_dir)`
5. 这是一个带会话、Agent、知识检索、记忆系统的项目，不是轻量静态页面，因此冷启动天然会更慢一些。

## 3. 关于 plugin 或 skill 的结论

- `Browser Use` 适合做浏览器侧验证、点击、截图、流程检查。
- 但它**不会直接提升项目启动速度**。
- 如果问题出在本地服务启动、编译、初始化，优先优化项目启动流程，而不是先增加 plugin 或 skill。

## 4. 后续建议

### 4.1 启动测试前先确认服务状态
在做前端联调或浏览器测试前，优先确认：

1. 前端 `3000` 是否已启动
2. 后端 `8004` 是否已启动
3. 后端 `/health` 是否返回正常

### 4.2 优先做项目级启动脚本
如果后续频繁需要启动和联调，优先考虑补充以下项目级脚本，而不是先抽象 skill：

- `start-dev.ps1`
- `stop-dev.ps1`
- 健康检查脚本

这样可以减少重复冷启动、减少手工判断、提升前后端联调效率。

### 4.3 skill 的使用时机
只有在以下条件满足后，再考虑把经验抽成 skill：

1. 启动流程已经稳定
2. 这套流程会在多个项目中重复使用
3. 需要沉淀成跨项目复用能力，而不只是当前仓库的本地脚本

## 5. 当前推荐实践

对于这个项目，当前推荐顺序是：

1. 先确认前后端是否已经运行
2. 如未运行，按 `AGENTS.md` 中的方式手动启动
3. 启动成功后再进入浏览器做前端测试
4. 若启动仍慢，优先排查后端初始化和前端首次编译，而不是优先增加外部工具
