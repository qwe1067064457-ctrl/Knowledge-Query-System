# Intent Rule Lessons

## 1. 文档定位

这份文档专门沉淀 `intent` 规则层的阶段性经验教训。

它回答的是：

1. 为什么这轮规则层做了这么久
2. 这轮真正做成了什么
3. 这轮最大的经验和教训是什么
4. 为什么规则层现在应该收口，并转向 SFT 准备

这不是规则流水账，也不是评估脚本说明，而是一份面向后续决策的复盘文档。

---

## 2. 原因分类：为什么规则层做了这么久

### 2.1 链路复杂度，不是单条规则复杂度

我们做的不是单点分类，而是一整条链路：

1. `input`
2. `evidence`
3. `resolved`
4. `control`

所以每次规则调整都不是“命中了就结束”，还要继续看：

- evidence 有没有产出对的候选
- resolved 有没有收敛对
- control 有没有映射到对的执行流

这天然比单纯的关键词分类慢。

### 2.2 标签里混有大量系统策略

有些标签相对客观：

- `qa`
- `chat`
- `unsupported`

但有些标签本质上带有系统设计选择：

- `needs_clarification`
- `mixed`
- `route=rag/direct/agent`
- `mode=challenge`

所以这轮工作不是只在做识别，也是在做策略建模。

### 2.3 一开始就在攻最难的边界

这轮调优没有停留在 happy path，而是很早就进入：

- `fuzzy_qa`
- `long_case_complex`
- `generic_qa`
- `judgment_qa`
- `challenge.soft_doubt`
- `meta boundary`

这些本来就是规则系统最容易掉分的区域。

### 2.4 后期已经不只是调规则，而是在补整套数据设施

真正耗时的部分后来越来越多地变成：

- `overall / per_batch / rule_stats`
- 规则级监督
- gold 数据
- held-out 数据
- frozen held-out
- 训练集导出
- SFT 准备文档

所以表面在调规则，实际是在建设一个可迁移的 intent baseline 系统。

---

## 3. 产出分类：这轮真正做成了什么

### 3.1 从“规则混合池”变成了分层 intent 系统

这轮逐步把 intent 链路讲清楚了：

- `input`
- `evidence`
- `resolved`
- `control`

也明确了：

- `signal_buckets`
- `ContextSignals`
- `global stable rules`
- `group_shared / domain bootstrap rules`

### 3.2 从“凭感觉调”变成“可评估调”

后面不再只是看感觉，而是明确依赖：

- `overall`
- `per_batch`
- `rule_stats`

这样才能知道问题到底在：

- evidence
- resolved
- control

中的哪一层。

### 3.3 从“看 hits”变成“做规则级监督”

后面最重要的一次升级是：

- 不再只看 `hits`
- 开始补 `expected_positive / expected_negative`

这让 `generic / judgment / soft_doubt` 终于能进入严格质量评估。

### 3.4 从“当前样本调优”变成“区分 dev / held-out / frozen held-out”

这一步非常关键，因为它避免了：

- 在同一批样本上越调分越高
- 但对真实泛化越来越失真

---

## 4. 经验分类：这轮最重要的经验

### 4.1 规则工程首先是建模问题，不是 regex 问题

最耗时的工作不是“哪个词要不要加”，而是这些问题：

- 什么是 `intent` 信号
- 什么是 `task` 信号
- 什么是 `context` 信号
- 哪些是强信号
- 哪些只是弱信号
- 什么应该在 evidence 决定
- 什么应该交给 resolver 收敛

所以规则层最难的首先是语义分层。

### 4.2 上游证据不稳，下游越调越像补锅

这轮反复验证了：

- evidence 没产出对的候选
- resolver 只能在错料里收敛
- control 只能稳定执行错误映射

因此 evidence 才是主战场。

### 4.3 高分不等于真实泛化

这轮多次出现：

- 小监督集上能打高分
- 一换 gold / held-out 就掉下来

这迫使我们真正区分：

- calibration / dev
- held-out
- frozen held-out

### 4.4 专业词表有价值，但不该被误当成长期架构

专业词表对冷启动有帮助，但长期问题也很明显：

- 不可扩展
- 和知识域耦合过深
- 容易滑向“词表补丁系统”

所以它更适合被定位成：

- `domain bootstrap`

而不是全局稳定语义。

### 4.5 真正值钱的不是单条规则，而是可迁移资产

这轮最应该保留的不是某条 regex，而是：

- 分层术语
- 信号分桶
- 监督口径
- gold / held-out / frozen held-out
- 导出格式
- SFT 准备文档

这些资产后面接小模型时还能继续用。

---

## 5. 教训分类：这轮最大的教训

### 5.1 规则级监督不能做得太晚

如果一条规则只有 `hits`，没有真值监督，那你只能看活跃度，不能看质量。

### 5.2 held-out 不能建得太晚

如果一直在同一批数据上调，分数会越来越好看，但泛化判断会越来越不可信。

### 5.3 不要把规则层目标设成“理解所有自然语言”

规则层适合做的是：

- baseline
- guardrail
- teacher
- data cleaning

它不适合承担最终开放语义理解。

### 5.4 文档如果不跟着收敛，后面会越来越难交接

没有文档，很难解释：

- 为什么某条规则这么写
- 为什么某个高分不代表真实泛化
- 为什么规则层现在应该收口

所以 `notes/intent/` 后面承担的不只是记录职责，也承担了认知降噪职责。

---

## 6. 收口分类：为什么规则层现在应该收口

当前最成熟的战略结论是：

- 规则层可以做 baseline
- 规则层可以做 guardrail
- 规则层可以做 teacher
- 规则层不该继续承担最终开放语义理解

这不是规则失败，而是规则边界被清楚识别出来了。

继续深调当然还能涨，但边际收益会越来越集中在小样本边界，维护成本会越来越高。

所以现在最合理的动作不是“无限继续调”，而是：

1. 保留规则层作为稳定基线
2. 固化监督和导出资产
3. 把下一阶段重点转向小模型 SFT

---

## 7. 对 SFT 的影响分类

这轮规则层工作为小模型准备直接留下了这些资产：

- 更清楚的标签空间
- 更清楚的层级字段
- 更可信的监督样本
- 更可解释的错误边界
- 更清楚的规则与模型职责分工

所以规则层花了很久，最终换来的不是“几条高分规则”，而是：

> 我们终于知道哪些事情应该继续交给规则，哪些事情应该转交给小模型。

---

## 8. 一句话总结

这轮规则层最成功的地方，不是“把规则调到多少分”，而是把这件事从“写几条正则”做成了：

> 一个有分层结构、有评估口径、有监督资产、能为小模型提供 teacher 和 guardrail 的 intent baseline 系统。
