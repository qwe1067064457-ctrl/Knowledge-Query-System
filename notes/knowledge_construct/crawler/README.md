# Crawler 经验记录

最后整理：2026-05-12

本文记录知识库采集过程中的爬虫经验，重点保留试错信息，避免后续重复踩坑。

## 1. 目标边界

这条线的目标不是“尽可能多抓页面”，而是：

- 优先保留原始文件
  - `pdf`
  - `docx`
  - 原始 `html`
- 如果拿不到原始文件，就不要把门面页当成知识正文
- 如果站点只能返回安全验证页、壳页面、占位页面，则应跳过或降级处理

核心原则：

1. 原文件优先
2. 抓不到原文件就跳过
3. 不把搜索结果页当知识库正文
4. 不把防护页误当成功下载

## 2. 已实践来源与结论

### 2.1 中国法律来源

#### FLK（国家法律法规数据库）

实践结论：

- 可以抓到一部分真实附件
  - `pdf`
  - `docx`
- 但并不是所有详情页都能稳定拿到原始文件
- 有些下载接口会返回反爬页，而不是文件流

关键教训：

- 不能因为接口返回 `200` 就认为下载成功
- 必须检查：
  - `content-type`
  - 文件头
  - 实际响应内容

典型失败现象：

- 请求下载接口后返回 `text/html`
- 页面标题类似“浏览器环境检查中”

经验结论：

- `FLK` 只能保留真实拿到的 `pdf/docx`
- 仅有 `meta.json` 或仅有详情页目录，不算完成采集

#### 最高法指导性案例

实践结论：

- 可以较稳定保留原始 `html`
- 适合作为法律知识库中的网页原文来源

注意点：

- `html` 原文可保留
- 后续如果要做可检索正文层，可以单独抽取，不应覆盖原始页面

#### 中国民商法律网

实践结论：

- 可作为民商事方向的原始 `html` 来源
- 适合补充民商事、公司法、合同、投融资等内容

### 2.2 医学来源

#### Yiigle

实践结论：

- 文章页中可拿到结构化线索
- 一部分页面可从内嵌数据中抽取较完整正文
- 但 PDF 直链经常被安全验证拦截

典型失败现象：

- 命中 PDF 线索链接
- 直连返回 `401`
- 响应不是 PDF，而是“安全验证”页面

经验结论：

- `Yiigle` 中“PDF 线索存在”不等于“PDF 可下载”
- 应区分：
  - 已发现 PDF URL
  - 已成功落盘 PDF
- 如果只能拿到原始页面和正文抽取层，就明确标记为“无 PDF 原文”

#### Nature / Nature Medicine

实践结论：

- 可以拿到公开文章的原始 `html`
- 但不少页面只是展示壳，不一定天然适合作为知识正文
- PDF 不是每篇都能稳定获得

经验结论：

- 不能把页面壳直接当成可用知识
- 需要区分：
  - 原始 `html`
  - 可用正文层
  - 是否存在可下载 `pdf`

#### IEEE

实践结论：

- 当前直连测试里遇到过 `418`
- 不适合当稳定原文主源

经验结论：

- 除非后续确认有稳定合规入口，否则不作为主采集源

#### Google Scholar / EasyScholar

实践结论：

- 适合作为检索入口
- 不适合作为原始文献仓库

经验结论：

- 只能用于发现候选文章
- 不应保存 Scholar / EasyScholar 搜索页作为知识库正文

#### PMC Open Access Subset / PMC FTP Service

实践结论：

- 是医学 PDF 原文最稳定的一条链路之一
- 可以从官方 CSV 清单定位 PDF
- PDF 是论文正文，不是门面页

这是当前最值得继续扩量的来源。

## 3. PMC FTP 这条链路的关键试错记录

这部分最重要，因为它是本轮最典型的“能跑通，但首轮踩坑很多”的例子。

### 3.1 不是直接就跑通的

实际试错过程包括：

1. 先尝试错误的 FTP 路径
2. 又尝试从 PMC 页面态入口找 PDF
3. 之后才确认应使用官方 CSV 清单
4. 再确认 CSV 实际可用路径在 `deprecated/` 下
5. 最后再修正 PDF 的 base URL

经验结论：

- PMC 这条线最靠谱的是“官方清单 + 官方 FTP 路径”
- 不要优先从页面层兜圈子

### 3.2 首版性能问题：每次都从 CSV 头部重扫

首版抓取器存在的问题：

- 每次运行都会重新打开大 CSV
- 从第一行开始扫描
- 即使前面的 PDF 已经存在，也要重新跳过一遍

这直接导致：

- 增量抓 1 到 2 份文档时也很慢
- 真正慢的不只是下载文件，而是前置扫描成本

### 3.3 已做的结构优化

后续已补的能力：

- 增加 `start_row`
- 在 `manifest.json` 中记录：
  - `start_row`
  - `next_row_offset`
- 下次运行默认优先从 `next_row_offset` 继续
- 对旧 manifest 增加兼容推断

这解决的是：

- “重复从头扫”的结构性问题

### 3.4 还没解决的问题

即使做了断点续抓，仍然存在一个现实限制：

- 当脚本真正去碰新的 FTP 区段时
- 远端网络链路本身仍然较慢

也就是说：

- 现在已经不是“逻辑上重复扫前面导致慢”
- 而是“官方远端链路本身慢”

这点要单独记住，避免后续误判为代码回退或算法失效。

## 4. 页面壳与知识正文的区分

这次一个很重要的经验是：

- `html` 文件存在
- 不等于“有知识数据”

应区分三层：

1. 原始层
   - `pdf`
   - `docx`
   - 原始 `html`
2. 正文层
   - 标题
   - 摘要
   - 正文
   - 作者
   - 日期
   - 关键词
3. 索引层
   - `index.md`
   - `manifest.json`
   - `metadata.json`

如果原始 `html` 只是壳页面：

- 不能把它误记为“已获得知识正文”

## 5. 当前可复用决策

后续再做新来源时，优先套用以下判断顺序：

1. 这个来源能否返回真实原始文件？
2. 如果是网页，网页里是否真有正文，而不是门面壳？
3. 是否存在反爬页、验证页、跳转页冒充成功响应？
4. 是否适合做批量增量抓取？
5. 是否值得做断点续抓？

建议优先级：

1. 官方开放获取 PDF 源
   - PMC FTP
   - Europe PMC
   - PLOS
   - medRxiv / bioRxiv
2. 可稳定抓原始 `html` 的站点
3. 只作检索入口的站点
4. 高反爬、不稳定、只能拿壳页的站点

## 6. 当前结论

截至目前，最稳定的经验总结是：

- 法律：保留真实 `pdf/docx/html`，抓不到原文件就跳过
- 医学：`PMC FTP` 是 PDF 主源，`Yiigle` 更适合作为“页面 + 正文抽取”来源，不适合作为稳定 PDF 主源
- 搜索站：只做发现，不做正文来源
- 任何来源都不要把门面页、验证页、空壳目录误判为完成采集
