# Knowledge 构建记录

最后整理：2026-05-12

本文记录知识库本身如何组织、如何落盘、如何分组与维护索引。

## 1. 目标边界

这里的“知识库构建”不是简单下载文件，而是把采集结果整理成后续 agent 可判断、可检索、可维护的结构。

目标包括：

- 目录结构清晰
- 原始文件保留稳定
- 每个来源有独立入口
- 索引可读且与真实文件状态一致
- 后续扩量时不破坏旧结构

## 2. 当前总原则

### 2.1 原文件优先

优先保留：

- `pdf`
- `docx`
- 原始 `html`

不要一上来把所有内容强行转换成同一种格式覆盖原件。

### 2.2 原始层与正文层分离

如果后续需要做抽取，应分层保存：

1. 原始层
   - `source.pdf`
   - `source.docx`
   - `source.html`
2. 正文层
   - `content.md`
   - `content.txt`
3. 元数据层
   - `metadata.json`
   - `manifest.json`
   - `index.md`

这样做的好处：

- 原文不会丢
- 抽取策略可以迭代
- 后续纠错时可回溯

### 2.3 抓不到原文就不要伪造完成态

以下内容不应算“知识库已完成落库”：

- 只有 `meta.json`
- 只有来源 URL
- 只有门面 `html`
- 只有搜索结果页
- 只有失败下载记录

## 3. 当前目录组织思路

知识库当前是按 group 组织的。

示意：

```text
backend/knowledge/groups/
  law/
    documents/
      cn/
      us/
  medicine/
    documents/
      ...
```

这层组织说明：

- `group` 是一级语义边界
  - `law`
  - `medicine`
- `documents` 放文献与知识材料
- 再按国家、来源、主题细分

## 4. law 组的构建经验

### 4.1 国家层拆分是合理的

法律资料实践中已经出现明显国家差异：

- 中国法律资料
- 美国法规资料

因此 `law/documents/` 下按 `cn/`、`us/` 继续拆是合理的。

### 4.2 是否继续深分组

当前更适合的方向不是无限细分 group，而是：

- `group` 维持较稳定
- 具体来源与分类更多交给索引和目录结构

原因：

- 过细分 group 会带来管理成本
- 而来源、专题、分类本身更适合交给 `index.md` 管理

### 4.3 cases 的处理

像 `cases` 这类共享内容会持续增长，不适合在 group 层再不断裂变小组。

更稳的做法是：

- 保持在组共享语义下
- 通过索引维护来源、年份、主题、标签

## 5. medicine 组的构建经验

### 5.1 医学知识库不能只保留页面壳

这次已经明确验证过：

- 某些原始 `html` 只有门面意义
- 并不能直接拿来做知识检索

所以医学组更适合两条路线：

1. 直接保留有知识密度的 `pdf`
2. 对网页类来源补正文层

### 5.2 PDF 主源与网页主源应分开认知

当前经验上：

- `PMC FTP` 更适合作为 PDF 主源
- `Yiigle` 更适合作为页面与正文抽取来源

落盘时不应把两者混为一个语义。

## 6. 索引文件规范

### 6.1 每个来源目录都应有 `index.md`

`index.md` 的作用不是装饰，而是：

- 说明来源
- 说明文档类型
- 记录当前数量
- 记录是否有正文层
- 说明使用限制或采集注意事项

建议至少包含：

1. 来源名称
2. 资料类型
3. 数量统计
4. 是否为原始文件
5. 是否有正文抽取层
6. 特殊限制说明

### 6.2 索引必须和真实目录状态一致

这是一个很重要的规则。

不能出现：

- 索引显示 0，目录里其实有附件
- 索引说有 PDF，实际上只有线索没有文件
- 索引说正文完整，实际上只有门面页

### 6.3 编码要稳定

之前已经出现过 `index.md` 中文乱码问题。

因此后续要尽量统一：

- 使用稳定编码写入
- 写完后实际检查可读性

如果索引文件可见内容已经乱码，即使结构在，也不算“符合规范”。

## 7. 元数据文件规范

### 7.1 `metadata.json`

适合单文档目录下使用。

建议记录：

- 来源
- 标题
- 日期 / 年份
- 原始 URL
- 文件 URL
- 分类
- 许可证或权限信息

### 7.2 `manifest.json`

适合来源目录级别使用。

建议记录：

- 来源名称
- 生成时间
- 总文档数
- 本轮写入数
- 跳过数
- 增量抓取偏移

例如 `PMC FTP` 已经把增量状态写成：

- `start_row`
- `next_row_offset`

这是后续增量抓取的重要基础。

## 8. 分类策略

分类要服务检索与后续 agent 判断，而不是为了把目录切得过碎。

当前更推荐：

1. group 维持稳定
2. 来源单独成目录
3. 文档内保留 `metadata.json`
4. 来源级保留 `index.md` / `manifest.json`
5. 如需更细主题分类，优先写入索引或元数据，而不是继续拆太多层级目录

## 9. 当前可复用落盘模板

单文档目录可参考：

```text
{record_dir}/
  source.pdf | source.docx | source.html
  content.md                # 如果有正文抽取层
  metadata.json
```

来源目录可参考：

```text
{source_root}/
  index.md
  manifest.json
  {record_dir_1}/
  {record_dir_2}/
```

## 10. 当前结论

知识库构建最关键的不是“抓到多少条”，而是：

- 原文件是否真实存在
- 索引是否可信
- 结构是否可持续扩展
- 后续 agent 是否能快速判断该目录里到底是什么

因此后续继续扩量时，优先坚持以下四条：

1. 原文优先
2. 结构分层
3. 索引一致
4. 抓不到就跳过，不伪造完成态
